# Check In Control

一个用于远程管理钉钉打卡规则的单用户 Web 控制台。

核心思路很简单：

- Web 端维护“周规则 + 日覆盖规则”
- iPhone 端快捷指令在执行打卡前先读取当天最终结果
- 网络失败时，iPhone 端仍然可以按本地兜底逻辑执行

适合“打卡手机长期放在固定地点，由快捷指令自动执行”的场景。

## 规则模型

系统只处理一件事：**今天最终是否打卡**。

判定优先级：

1. 日规则
2. 周规则
3. 默认不打卡

举例：

- 周四默认打卡，如果当天没有日规则，结果就是打卡
- 周四默认打卡，如果当天日规则强制不打卡，结果就是不打卡
- 周一默认不打卡，如果当天日规则强制打卡，结果就是打卡

所有日期和星期判断固定使用 `Asia/Shanghai`，避免服务器时区和手机时区不一致导致误判。

## 当前功能

- 单管理员账号
- 首次默认密码登录后强制修改密码
- 周规则管理
- 日历视图管理未来日期
- 日规则覆盖周规则
- 生成带随机 `api_key` 的接口地址供快捷指令调用
- `/api/checkin/today` 返回纯文本 `true` / `false`
- 页面支持响应式布局，桌面和手机共用一套界面结构

## Web 控制台

当前 Web 控制台负责：

- 配置每周默认打卡规则
- 查看整个月的最终打卡状态
- 直接切换某一天最终是否打卡
- 生成并轮换快捷指令使用的 API 地址

## 技术栈

- `Next.js` App Router
- `Postgres`（推荐 `Supabase`）
- `Vercel`
- 原生 `crypto.scrypt` 做密码散列
- HMAC Cookie Session

## 环境变量

项目依赖以下环境变量：

- `DATABASE_URL`
- `DATABASE_SSL`
- `DEFAULT_ADMIN_PASSWORD`
- `SESSION_SECRET`
- `NEXT_PUBLIC_APP_URL`

参考模板见 [.env.example](./.env.example)。

## 本地运行

1. 复制环境变量模板

```bash
cp .env.example .env.local
```

2. 填写数据库、密码、会话密钥和站点地址

3. 安装依赖并启动

```bash
npm install
npm run dev
```

首次运行会自动初始化数据库内容：

- 单管理员账号
- 固定时区 `Asia/Shanghai`
- 随机 `api_key`
- 默认周规则：周四到周日打卡

## 部署

推荐部署方式：

- 代码托管到 GitHub
- 应用部署到 Vercel
- 数据库使用 Supabase Postgres

部署步骤简述：

1. 在 Supabase 创建 Postgres 数据库
2. 在 Vercel 导入本仓库
3. 配置环境变量
4. 首次部署完成后访问 `/login`
5. 使用 `DEFAULT_ADMIN_PASSWORD` 登录并立即修改密码

如果已经绑定自定义域名，记得同步更新 `NEXT_PUBLIC_APP_URL`，否则后台里显示的快捷指令地址会还是旧域名。

## 快捷指令接口

接口地址：

```txt
GET /api/checkin/today?key=<api_key>
```

成功时返回纯文本：

```txt
true
```

或：

```txt
false
```

未授权时返回：

```json
{ "error": "unauthorized" }
```

对 iOS 快捷指令来说，纯文本 `true` / `false` 比 JSON 更容易判断。

## iPhone 快捷指令方案

当前推荐的是“双快捷指令 + 自动化”方案。

### 1. 同步今日计划

用途：

- 调用 Web API
- 读取今天最终是否打卡
- 把结果缓存到手机本地可读取的位置，供后续真正执行打卡的快捷指令使用

截图：

![同步今日计划](./shortcut/同步今日计划.png)

### 2. 智能打卡

用途：

- 读取同步结果
- 如果结果为 `true`，继续执行钉钉打卡
- 如果结果为 `false`，直接结束
- 如果同步失败或没有缓存，按本地兜底逻辑执行

截图：

![智能打卡](./shortcut/智能打卡.png)

### 3. 强制打卡

用途：

- 作为人工兜底入口
- 在需要时手动触发，不经过自动规则判断

截图：

![强制打卡](./shortcut/强制打卡.png)

### 4. 自动化

用途：

- 在固定时间自动执行快捷指令
- 例如早上和傍晚分别跑一次

截图：

![自动化](./shortcut/自动化.PNG)

## 推荐执行链路

建议 iPhone 端按下面的顺序组织：

1. 自动化触发“同步今日计划”
2. 自动化再触发“智能打卡”
3. “智能打卡”读取同步结果决定是否打开钉钉
4. 若网络失败，则退回到本地预设的周规则兜底逻辑

这样做的好处是：

- Web 端可以随时改当天是否打卡
- iPhone 端仍保留离线兜底
- 快捷指令判断逻辑更简单，排查也更直接

## 备注

- 当前系统是单用户设计，不支持多账号
- 当前系统按“整天是否打卡”处理，不区分早上/晚上两套独立规则
- 页面中的日历单元格交互语义是“设置当天最终是否打卡”，不是直接编辑底层日规则对象
